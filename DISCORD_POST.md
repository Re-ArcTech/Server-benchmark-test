📡 **通信方式の検証おわったよ（Ver.3.0サーバー）**

Photonをやめて独自サーバーにするにあたり、繋ぎ方の候補 **REST / WebSocket / WebRTC** を全部作って同じ条件で競争させました。結論から:

**■ 結論**
> ・ゲーム中の同期 → **WebSocket**（本命）
> ・ログイン/ランキング → **REST**（普通のWeb通信）
> ・WebRTC → 当面なし（電波が悪い時の将来の保険）
> ・サーバーの人数心配は不要（1台1000人で余裕＝Photon無料枠の50倍）

**■ 何をやった/なぜ**
3つの候補それぞれの得意・不得意を、実際のゲームのやり取り11種類（移動/キック/ゴール/ログイン/マッチング/同期/タイマー…）で計測。

**■ どこで試したか（近い→遠い）**
```
ローカル(PC内) → サーバー処理の重さを測る
Render(海外)   → 往復 約100ms
AWS東京        → 往復 約15ms ← 国内は爆速
```
→ 本番サーバーは国内(東京)に置くべき、と確定。

**■ 負荷テスト（一番知りたかった所）**
1台で **1000人つないでもCPU20%/メモリ50MB** で余裕。
Photonの「20人まで」は技術の限界じゃなくお金の都合だった。独自サーバーなら桁違いにさばける💪

**■ わざと電波を悪くして試した（クライマックス）**
本番ユーザーはスマホ＝電波が乱れる。なので **わざと8%通信ロス** を起こして再計測。

たとえ:
> ・REST/WebSocket（宅配便）＝順番通り全部届けるが、1個無くなると後ろが待つ → **たまに最大2秒フリーズ😱**
> ・WebRTC（ドローン投下）＝投げっぱなし。無くなっても次へ → **滑らかだが2割捨てる✨**

→ 位置情報なら「2秒フリーズ」より「滑らかで一部欠け」が断然マシ（最新位置だけ欲しいので）。
**電波が悪い時"だけ"WebRTCに価値あり**、と実証できた。

**■ 「同期」は2種類あるの大事**
> ・ゴール/スコア/必達イベント → **WebSocket**（絶対届ける）
> ・位置/ボールの動き → 将来は **WebRTC**（最新優先・欠落OK）
だから「まずWS一本 → モバイルのカクつきが問題化したら位置だけWebRTC」。

**■ サーバーの置き場（CI/CDで常時稼働は？）**
本番ではまさにそうする（クラウド常時稼働＋自動デプロイ＋HTTPS）。発想は正しい。
ただ常時稼働が活きるのは本番から。開発中はアイドルに金払うだけ損なので「使う時だけ起動 or Render無料(寝てタダ)」でいく。

**■ クライアント側で意識してほしいこと**
> ・相手の動きは**補間で滑らかに**見せる前提で（受信間隔はバラつく）
> ・「必達(ゴール/スコア)」と「最新優先(位置)」を分けて考える
> ・接続は張りっぱなし基本（再接続は約350msと地味に重い）
