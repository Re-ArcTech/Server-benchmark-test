# 通信方式ベンチ 計測結果まとめ

ゆびサッカー Ver.3.0 サーバー方式検証。REST / WebSocket / WebRTC を、速度だけでなく
**スケール性・サーバー負荷・耐ロス性・再接続・帯域**まで含めて比較した。

計測日: 2026-06-13 / サーバー: Go (gorilla/websocket) / クライアント: Go loadtest + Unity

## 計測環境の前提（数字を読む前に）

- **ローカル(localhost)** = ループバック。ネットワーク遅延ほぼ0なので **サーバー処理コストそのもの** が見える。
- **Render** = 日本 → Render(シンガポール無料枠)。**実ネットワークの往復(~100ms)** が支配的。
- 国内(東京リージョン)なら往復は **20〜40ms** に下がる見込み（Renderに日本リージョンが無いだけ）。
- RTTはクライアントの単一時計(Stopwatch)で計測。`serverProc = serverSend - serverRecv` でサーバー内処理を分離。

---

## 1. 速度（RTT）— 実ネットワーク(Render)

全シナリオ avg(ms) / p95(ms) / jitter(ms)。往復~100msが下限。

| シナリオ | WS avg | WS p95 | WS jit | REST avg | REST p95 | REST jit |
|---|---|---|---|---|---|---|
| echo(ping) | 105 | 158 | 26 | 126 | 236 | 53 |
| player.move | 105 | 155 | 25 | 105 | 148 | 20 |
| ball.kick | 105 | 155 | 25 | 105 | 148 | 20 |
| goal.check | 105 | 155 | 25 | 109 | 159 | 30 |
| state.sync(1.5KB) | 105 | 153 | 22 | 109 | 154 | 27 |
| asset.load(64KB) | **114** | 177 | 40 | **144** | 203 | 38 |

**読み取り:**
- 接続維持後の毎メッセージは **WS ≒ REST(keep-alive)** だが、**REST はテール(p95)とジッターが大きい**（echo p95 236ms vs WS 158ms）。リアルタイム同期では「たまに大きく遅れる」が致命的なので **WSが有利**。
- **大きいデータ(asset.load 64KB)はWSが明確に速い**（114 vs 144ms）。
- サーバー処理は実ネットでは誤差（下記2参照）。**遅延はほぼ純ネットワーク**。

### Unityクライアント実測(Render, フレーム量子化是正後)
| | connect | move | kick | goal |
|---|---|---|---|---|
| REST | 311ms | 156 | 159 | 157 |
| WebSocket | 695ms | 108 | 130 | 125 |

→ Unityでも **WSが毎メッセージ30〜50ms速い**。接続確立はWSの方が重い(一度きり)。

---

## 2. サーバー処理コスト（type別, ローカル=純処理）

`serverProc` の平均(ms)。「どのやり取りがサーバーCPUを食うか」。

| シナリオ | srvProc | 備考 |
|---|---|---|
| **auth.login** | **1.03ms** | KDF(sha256 x1万) + JWT発行。**本物のbcrypt/argon2なら50〜100ms級** |
| asset.load(64KB) | 0.19ms | データ配信 |
| ball.kick | 0.18ms | 擬似物理(60step) |
| auth.verify | 0.05ms | JWT検証 |
| state.sync | 0.04ms | 6人+3球スナップショット生成 |
| goal.check / match / room / timer / move / echo | <0.02ms | ほぼ無料 |

**読み取り:**
- **認証(login)だけが突出して重い。** ゲーム中のメッセージ(move/kick/sync)はサーバーCPUをほぼ使わない。
- → 設計示唆: **ログインは別経路(REST) + レート制限 + 重いKDFはログイン時のみ**。ゲームループにauthを混ぜない。

## 3. スケール性（CCU）— 最重要

「1台で何人さばけるか」。RTT劣化とサーバーリソース。

### ローカル(8コアMac, 各5msg/s)
| 同時接続 | RTT avg | RTT p95 | サーバーCPU | メモリ | goroutine |
|---|---|---|---|---|---|
| 50 | 0.35ms | 0.55ms | ~5% | 15MB | 53 |
| 200 | 1.78ms | 2.69ms | ~9% | 16MB | 203 |
| **1000** | **0.18ms** | **0.37ms** | **~20%** | **50MB** | 1003 |

### Render無料枠(CPU極小, 各3msg/s)
| 同時接続 | RTT avg | RTT p95 | メモリ |
|---|---|---|---|
| 50 | 105ms | 166ms | 13MB |
| 150 | **105ms** | **167ms** | 17MB |

**読み取り（独自サーバー判断の核）:**
- **サーバーはボトルネックでない。** ローカルで1000接続=CPU20%/50MB。**Render無料枠ですら150接続でRTT劣化ゼロ**（遅延は純ネットワークのまま）。
- **Photonの無料20CCU制限は技術的ではなく商用的な縛り。** 独自サーバーなら1台で数千接続が現実的。
- → サーバー権威も「処理が軽い」ので十分射程内（重いのは認証くらい）。

---

## 4. 耐ロス性（TCP HOLブロッキング実証）— WebRTC検討の核

250msのストール(=パケットロス再送待ち)を2秒ごとに注入した時のWS(TCP)の挙動。

| | avg | p99 | max | 50ms超 |
|---|---|---|---|---|
| 直結(ロス無) | 0.33ms | 1.31ms | 3ms | 0% |
| **ストール注入** | 4.63ms | **229ms** | 251ms | 2% |

**読み取り:**
- **ロスが起きるとTCP/WebSocketは後続メッセージが巻き添えで詰まる**（head-of-line blocking）。p99が229msに跳ねる。
- **UDPベースのWebRTC(unreliable)ならロスしたパケットだけ捨てて後続は流れる** → 巻き添えが起きない。
- → **WebRTCの唯一の存在意義はここ。** 良回線ではWS≒WebRTCなので、**ロスの多いモバイル回線の位置同期(取りこぼし許容)に限ってWebRTCが効く**。それ以外はWSで十分。
- ※実機セルラーでの実ロス計測はNetwork Link Conditioner(要sudo)＋UDP対応VPSでのWebRTC計測が次ステップ。

## 5. 再接続コスト

| | ローカル | Render(実ネット) |
|---|---|---|
| connect(TCP+TLS+WS) | 1.05ms | **353ms** |
| 復帰後の初メッセージ | 0.12ms | 92ms |

**読み取り:**
- **実ネットの再接続は約350ms**（TLSハンドシェイクの往復ぶん）。決して無料でない。
- → 設計示唆: **接続は張りっぱなしを基本に。** 試合中の切断→再接続は350ms+状態復元のコスト。WS常時接続が有利。

## 6. 帯域（ペイロードサイズ依存）

WebSocket echo.size, Render(実ネット) avg。

| サイズ | RTT | 増分 |
|---|---|---|
| 64B | 109ms | 基準 |
| 1KB | 103ms | ~0 |
| 4KB | 107ms | ~0 |
| 16KB | 113ms | +4ms |
| 64KB | **149ms** | **+40ms** |

**読み取り:**
- **16KBまでは実質タダ**（パケットに収まる）。**64KBで+40ms**。
- → 設計示唆: Ver.3.0の「ボール複数化・全体同期スナップショット」は **16KB以内に収める**(差分圧縮)。state.syncの6人+3球=1.5KBは余裕。

## 7. 連続送信（fire-and-forget, 30fps×10s）

返事を待たず撒き続けた時の到着状況。

| | ローカル | Render(実ネット) |
|---|---|---|
| 取りこぼし | 0/300 | 0/300 |
| 平均到着間隔 | 33.3ms | 33.3ms |
| 間隔ジッター | 0.33ms | **22ms** |
| 最大間隔 | 34.6ms | 122ms |

**読み取り:**
- ロスは0だが、**実ネットでは到着間隔が揺れる**(jitter 22ms, 最大122ms)。
- → クライアント側の**補間(interpolation)は必須**。受信間隔がばらつく前提で滑らかに見せる。

## 速度以外も含めた「方式ごとの強み」総まとめ

| 観点 | REST | WebSocket | WebRTC(DataChannel) |
|---|---|---|---|
| 毎メッセージ遅延 | △(テール大) | ◎ | ◎ |
| ジッター安定性 | △ | ◎ | ◎ |
| サーバーからのpush | ✕(不可) | ◎ | ◎ |
| 大データ配信 | △ | ○ | ○ |
| 同時接続スケール | ○ | ◎(1台数千) | △(P2P/TURN要) |
| 耐パケットロス | ✕(HOL詰まり) | ✕(HOL詰まり) | ◎(巻き添え無) |
| 実装の簡単さ | ◎ | ○ | ✕(signaling/ICE/TURN) |
| Renderでデプロイ | ◎ | ◎ | ✕(UDP不可) |
| ステートレス性 | ◎(キャッシュ可) | △(接続保持) | △ |
| **向いてる用途** | ログイン/プロフィール/ランキング | **ゲーム中の同期全般** | ロス回線の位置同期(取りこぼし許容)のみ |

## Ver.3.0 への結論

1. **ゲーム中の同期は WebSocket。** 毎メッセージのテール/ジッターが小さく、push可能で、1台で数千接続さばける。
2. **ログイン/プロフィール/ランキングは REST。** リクエスト単発・キャッシュ可・ステートレス。**認証KDFは重い(本物50-100ms)のでレート制限必須**。
3. **WebRTCは当面不要。** 効くのは「ロスの多いモバイル回線の位置同期」だけ。複雑さ(signaling/ICE/TURN)とRenderで動かない制約に見合うのは、実機セルラーで実ロス由来のカクつきが問題化してから。
4. **サーバー権威は射程内。** ゲームメッセージのサーバー処理はほぼ無料。重いのは認証だけ。Photonの20CCUは商用縛りで、独自サーバーなら桁違いにさばける。
5. **接続は張りっぱなし前提。** 実ネット再接続は~350ms。切断ハンドリングは「いかに再接続を避けるか/状態を保つか」が肝。
6. **同期データは16KB以内＋クライアント補間必須。** 帯域の壁は64KB付近、実ネットは到着間隔が揺れる。

## 再現方法

```bash
# サーバー（ローカル）
cd server && go run .

# 各計測（loadtestツール）
go run ./cmd/loadtest scenarios -url http://localhost:8080 -n 20
go run ./cmd/loadtest ccu       -url ws://localhost:8080/ws -conns 1000 -duration 10s
go run ./cmd/loadtest stall     -upstream localhost:8080          # HOLブロッキング実証
go run ./cmd/loadtest sizes     -url http://localhost:8080
go run ./cmd/loadtest reconnect -url ws://localhost:8080/ws -times 15
go run ./cmd/loadtest stream    -url ws://localhost:8080/ws -fps 30 -seconds 10

# Render(実ネット)は -url https://yubi-bench-server.onrender.com に変えるだけ
# Unity: BenchScene.unity を開いて Play → 計測スタート（既定でRenderを向く）
```
