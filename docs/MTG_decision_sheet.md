## 2. 権威モデル（「サーバーは"何の"権威を持つか」で分解する）

「サーバー権威 vs クライアント権威」を塊で悩まない。**対象ごとに分けると迷うのは1つだけ**。

| 対象 | 推奨 | 備考 |
|---|---|---|
| 試合進行（開始/終了/タイマー） | **サーバー権威** | Photonのマスター廃止。サーバーが回す。軽い |
| ゴール判定・スコア・勝敗 | **サーバー権威** | ベンチで~0ms実証。レート制の前提。改ざん不可 |
| 入力・移動（プレイヤー位置） | **クライアント送信＋補間** | サーバー全権はやり過ぎ |
| **ボール物理** | **クライアント権威** | ★唯一の悩み所。下記 |

**ボール物理の推奨: クライアント権威で確定（基本"2回実装"はしない）**
- 理由: ゴール/スコアは既にサーバー権威なので、ボール位置を改ざんされても**点は増やせない**＝チートの実害が小さい → 物理をサーバーに移す動機がほぼ生まれない
- 万一移す時も、権限判定を1箇所に集約しておけば**ボール同期コードだけ**の差し替えで済む（全面書き直し回避）
- おまけ: 物理がクライアント側なら**サーバーは重いtickループ不要**（イベント駆動＋中継で済む＝工数読みやすい）
- 決定: ☐ __________

# 🗓 MTGの進め方

## 事前準備（前日まで）
- このシートを全員に共有し、**各自ざっと目を通してくる**（特にカシューはプロトコル＝クライアントとの契約なので要確認）
- りーせ＝ファシリテーター（進行）、書記を1人決める（決定欄をその場で埋める）

## 当日の流れ（目安60〜75分）
```
00-05分  ゴール確認「このシートの決定欄を全部埋める」
05-25分  1. プロトコル（最重要・最も時間を割く。メッセージ一覧の抜け確認）
25-35分  2. 権威モデル（分解の整理を見せて、ボール物理だけ合意）
35-45分  3-4. 認証・マッチング
45-55分  5-6. 試合ルール・再接続
55-65分  7-9. 同期・後回し項目・抜け項目（推奨どおりサクサク確認）
65-75分  保留(🔺)になった項目の宿題割り当て・次アクション確認
```

## 決め方のルール（時間を溶かさないコツ）
1. **推奨をデフォルトにする。** 異論が無ければ即「✅」で次へ（全部ゼロから議論しない）
2. **5分揉めたら🔺保留にして次へ。** その場で結論が出ない物に粘らない。宿題化
3. **「プロトとして」を判断軸に。** 「将来は…」で広げない。まず動くものを作る前提で決める
4. **決定はその場で書記が決定欄に記入。** 口頭で終わらせない

## アウトプット
- 全項目の決定欄が埋まったシート（＝そのまま実装の指示書になる）
- 🔺保留項目の宿題リスト（担当・期限つき）
- 次アクション（6月末までに各自やること）

## 揉めそうな所の事前対策
- **権威モデル**: 「2回実装が面倒」が論点になりがち → 2章の「分解すれば迷うのはボール物理だけ・移行は基本しない」を先に見せると早い
- **メッセージ一覧**: 抜けが必ず出る → 「今出なくても後から追加できる設計」と握っておけば、完璧を目指して止まらずに済む
