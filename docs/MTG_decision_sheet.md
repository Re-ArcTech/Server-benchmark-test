# Ver.3.0 サーバー仕様 決定シート（MTG叩き台）

木曜の決定MTG用。**各項目「推奨」をデフォルトにして、異論なければ即決→決定欄を埋める**。
依存関係のある順に並べてあるので、上から進めると手戻りしない。

> 凡例: 各項目に「推奨」あり。MTGでは ✅(推奨どおり) / 変更(内容を書く) / 🔺保留(宿題化) のどれかを決定欄に記入。

> 🧭 **設計原則（2026-06-19）**: チート対策とUXはトレードオフ。**迷ったらユーザーへの価値を優先**する。
> このゲームは「爽快感・わちゃわちゃ」が軸なので、厳密な審判より気持ちよさを取る。
> サーバーは"安全網"として効かせ、デフォルトの体験は気持ちよく倒す。

## 2. 権威モデル（2026-06-19 MTGで確定）

基本思想: **基本はクライアントで処理 → サーバーが承認/訂正（ハイブリッド）**。

| 対象 | 担当 | 補足 |
|---|---|---|
| 試合進行（開始/終了/タイマー） | サーバー権威 | Photonのマスター廃止。サーバーが唯一の権威 |
| **ゴール判定** | **クライアント検知 ＋ サーバー承認** | 見た瞬間に検知（見た目一致＝萎え対策）→ サーバーが妥当性チェックして確定 |
| スコア・勝敗 | サーバー権威 | サーバーが最終加算。レート制の前提 |
| プレイヤー移動 | **クライアント予測 ＋ サーバー訂正** | 自分の操作は即反映、ズレたらサーバーが直す |
| ボール | **クライアント処理 ＋ サーバー訂正** | 同上。基本クライアント、間違いはサーバーが修正 |

**決定: ✅ ゴール=クライアント検知＋サーバー承認 / 移動・ボール=クライアント予測＋サーバー訂正（ハイブリッド）**

補足・注意:
- **ゴールUX**: 「見えた瞬間に祝う」のではなく、サーバー承認後に確定演出（30-60msは気づかれない）。「盛り上げ(ローカル)」と「確定(サーバー)」を分ける。ゴール判定は厳密さより**気持ちよく**（広いゾーン＋吸い込み、迷ったら入れる）
- ⚠️ **工数注意**: 「サーバー訂正(ロールバック)」をフルで作ると重い（FPS向けの重い技術）。**まず予測＋補間まで作り、訂正は段階的に**入れるのを推奨。プロトで全部作らない
- ボールがクライアント側でも、サーバーは位置ストリームを受けてゴール承認＋サニティチェックできる（チートのハードルを上げる）

### 8-2. バージョン管理・互換性（★重要・最初から設計に入れる）
**過去バージョンのアプリ・過去サーバーにもユーザーが残る** → old/newが混在する前提で設計する。

- **接続時に `clientVersion` ＋ `protocolVersion` を送る**。サーバーは `minSupportedVersion` 未満を弾く（VERSION_MISMATCH → 強制アップデート促す）
- **メッセージは「追加のみ(additive)」で進化させる**: フィールドを消さない・リネームしない・新規はoptional。これでold/newが混ざっても壊れない（未知フィールドは無視する作りにする）
- **マッチングは同一 protocolVersion 同士のみ**（違うバージョンを混ぜると同期ズレ）→ バージョンをマッチングキーに含める
- 複数サーバーバージョンを併存させるなら、バージョンごとにルーティング（または1サーバーが複数protocolVersionを喋れるように）
- 決定: ✅ 最初から仕込む / 追加のみ進化 / バージョン別マッチング / minSupportedVersionで足切り

## 10. アセット配信（Addressable Assets）とセキュリティ（2026-06-19 確定）

| 種別 | 配信方法 | 審査 |
|---|---|---|
| ゲームロジック（スクリプト） | ネイティブ組み込み | 必要 |
| アセット（モデル/ステージ/BGM/画像/UI） | **Addressable Assets でサーバー配信** | 不要 |
| パラメーター（数値・設定値） | **JSON でサーバー管理** | 不要 |
| 新コンテンツ（新キャラ等） | ビルド同梱 → **サーバーフラグで公開制御** | 初回のみ |

※ Appleは「リモートのコードDL実行」を禁止 → **ロジックは必ず審査・アセットと設定値だけサーバー**。

### ★セキュリティ（IDの難読化）
アドレス/URL/カタログを生で置くと**簡単に盗まれる・未公開コンテンツがリーク**する。
- アセットバンドルを**暗号化**（Addressableのカスタムプロバイダ）、**カタログも暗号化/難読化**
- 配信は**署名付きURL（期限つき）**が理想 → URLを盗まれても期限切れで無効
- **未公開キャラ（フラグ非公開）はビルドに同梱されている**ので、暗号化しないと解析でリークする → 暗号化必須
- ⚠️ 完全防御は不可能（クライアントで復号する以上、鍵もクライアント側にある）。難読化は**「解析を遅らせる・カジュアルな盗用を防ぐ」割り切り**で考える
- 決定: ✅ Addressable + JSON / 暗号化・署名付きURL・難読化を前提に設計

# 🗓 MTGの進め方

## 事前準備（前日まで）
- このシートを全員に共有し、**各自ざっと目を通してくる**（特にカシューはプロトコル＝クライアントとの契約なので要確認）
- りーせ＝ファシリテーター（進行）、書記を1人決める（決定欄をその場で埋める）

## 当日の流れ（目安60〜75分）
```
00-05分  ゴール確認「このシートの決定欄を全部埋める」
05-25分  1. プロトコル（最重要・最も時間を割く。メッセージ一覧の抜け確認）
25-35分  2. 権威モデル（分解の整理を見せて、ボール物理だけ合意）
35-45分  3-4. 認証・マッチング
45-55分  5-6. 試合ルール・再接続
55-65分  7-9. 同期・後回し項目・抜け項目（推奨どおりサクサク確認）
65-75分  保留(🔺)になった項目の宿題割り当て・次アクション確認
```

## 決め方のルール（時間を溶かさないコツ）
1. **推奨をデフォルトにする。** 異論が無ければ即「✅」で次へ（全部ゼロから議論しない）
2. **5分揉めたら🔺保留にして次へ。** その場で結論が出ない物に粘らない。宿題化
3. **「プロトとして」を判断軸に。** 「将来は…」で広げない。まず動くものを作る前提で決める
4. **決定はその場で書記が決定欄に記入。** 口頭で終わらせない

## アウトプット
- 全項目の決定欄が埋まったシート（＝そのまま実装の指示書になる）
- 🔺保留項目の宿題リスト（担当・期限つき）
- 次アクション（6月末までに各自やること）

## 揉めそうな所の事前対策
- **権威モデル**: 「2回実装が面倒」が論点になりがち → 2章の「分解すれば迷うのはボール物理だけ・移行は基本しない」を先に見せると早い
- **メッセージ一覧**: 抜けが必ず出る → 「今出なくても後から追加できる設計」と握っておけば、完璧を目指して止まらずに済む
