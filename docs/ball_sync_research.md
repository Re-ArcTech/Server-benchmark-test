# ボール同期 調査メモ

プレイヤー位置の次の本丸＝**ボール（共有1個）をどう同期するか**の調査。
（リーダー依頼「同期調べて」の続き）

## 問い
プレイヤーは「各自が自分を動かす」で済む。だがボールは**1個を全員で共有**する。
誰が動かす？ 触ったらどう移る？ 接戦したら？

## 実例①：ロケットリーグ（物理ボール共有の代表例）
- **完全サーバー権威**。サーバーが固定tick(120Hz)でボール物理を回す
- **クライアントは全部（自車・他車・ボール）を予測**し、サーバーから訂正（client-side prediction + reconciliation）
- サーバーは再シミュレーションせず、入力バッファで遅延ムラを吸収。バッファが枯れたら直前入力を繰り返す
- → サーバーがボールを持つので**所有権の曖昧さがゼロ＝最もフェア**。代償は**サーバー物理が必要**

## 実例②：所有権の移譲（Unity Netcode distributed authority 等）
- ボールに**オーナー**を持たせ、触った人が所有→シミュレート→配信。別の人が触ると移譲
- 所有権モード: 即移譲(Transferable) / 要求制(RequestRequired) / 自動再配分(Distributable) など
- **★最大の難問＝接戦のレースコンディション**：2人が同時に取りに行くと所有が衝突。
  ロック(SetOwnershipLock)や所有権+権限を同一バッチで送る等で対処するが、ここが鬼門

## ★重要な気づき：ボールは「補間」でなく「外挿」
プレイヤーと真逆：
```
他プレイヤー → 補間（過去を描く） … 自分で操作しないので遅れてOK
ボール       → 外挿（未来を予測） … 当てに行くので過去位置だと打てない
```
ロケットリーグも「クライアントはボールも予測する」と明言。**100ms前のボールは当てられない**。
→ ボールは速度で先読みして「今ここにあるはず」を描く。設計で見落としやすい超重要点。

## ゆびサッカーへの結論
決定済み「ボール＝クライアント処理＋サーバー訂正」は**実例②（所有権移譲）に該当**：
- 触った人がボールを配信、別の人が蹴ったら所有権が移る
- サーバーは物理を回さないが「誰が所有か」を裁定＋ゴール判定＋必要なら訂正
- 他人のボールは**外挿（先読み）**で描く
- 接戦は「わちゃわちゃ軸」で甘く裁く（サーバーが最後に見たタッチを採用）
- **完全サーバー権威（ロケットリーグ式）は将来のアップグレード先**（接戦チート/ズレが問題化したら）

## 残る難問
**接戦時の所有権（2人同時タッチ）**。サーバーが裁定者になる必要がある（物理は回さなくても「誰が触ったか」は裁く）。
裁定ルール候補：サーバーが見た時刻順 / ボールへの距離 / 最後の有効タッチ。

## ラボで実験すべきこと（次アクション）
- ボールを1個追加し、「触った人が所有→配信」「他人は外挿で描画」を実装
- 所有権の移譲（キックで持ち主が変わる）を可視化
- 遅延/ジッター下で「接戦時にどう見えるか・どう裁定すれば自然か」を体感
- 外挿ON/OFFで「ボールが当てやすいか」を比較
